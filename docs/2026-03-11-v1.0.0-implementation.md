> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Harden and complete the minerva-deploy stack to achieve a working, reproducible GitOps deploy via GitHub Actions + Tailscale.

**Architecture:** Three chunks of work build on each other: (1) fix known bugs and critical/high security issues, (2) harden the CI/CD pipeline and add a Tailscale-based deploy workflow, (3) update docs and cleanup. Each chunk produces a tested, lint-passing commit.

**Tech Stack:** Ansible + Molecule (Docker driver), Docker Compose, Ubuntu 24.04, GitHub Actions, Tailscale.

### Task 4: Fix compose-sync Missing Notify Handler

**Files:**
- Modify: `ansible/roles/compose-sync/tasks/main.yml`
- Modify: `ansible/group_vars/all/all.yml`
- Modify: `ansible/roles/compose-sync/molecule/default/verify.yml`

The sync task has no `notify`, so compose.yml changes never restart containers. Additionally, `compose_sync_restart_services` is never set, so even `.env` change notifications restart nothing.

- [ ] **Step 1: Add `notify` to the sync task**

In `ansible/roles/compose-sync/tasks/main.yml`, add `notify: Restart affected services` to the `Sync compose directory to remote` task:

```yaml
- name: Sync compose directory to remote
  ansible.builtin.copy:
    src: "{{ compose_sync_src_dir }}/"
    dest: "{{ compose_sync_base_dir }}/compose/"
    owner: "{{ compose_sync_owner }}"
    group: "{{ compose_sync_group }}"
    mode: preserve
    directory_mode: "0755"
  notify: Restart affected services
```

Do NOT add `compose_sync_restart_services` to the Molecule converge.yml. The handler already uses `| default([])`, so when the variable is absent the handler runs without restarting anything — this is correct for the test environment which has no Docker daemon. Adding a service name to the converge vars would cause the handler's `docker_compose_v2` task to fail in the test environment.

- [ ] **Step 2: Wire `compose_sync_restart_services` in `all.yml`**

In `ansible/group_vars/all/all.yml`, add to the Role vars section (below `compose_up_base_dir`):

```yaml
compose_sync_restart_services: "{{ compose_up_services }}"
```

This means: whenever any compose file or `.env` changes, all auto-started services are restarted. Acceptable for a single-server homelab GitOps deploy.

- [ ] **Step 3: Run compose-sync Molecule test**

```bash
cd ansible/roles/compose-sync && molecule test
```

Expected: PASS. The handler will be notified but the `| default([])` in the handler loop means no `docker_compose_v2` calls run in the test environment.

- [ ] **Step 4: Commit**


GitHub Actions use annotated tags, so `git/ref/tags/v4` returns a tag object SHA, not a commit SHA. Dereference it with a second call:

```bash
# Helper function: dereference annotated tag to commit SHA
get_action_sha() {
  local repo=$1 tag=$2
  local tag_sha type
  tag_sha=$(gh api "repos/$repo/git/ref/tags/$tag" --jq '.object.sha')
  type=$(gh api "repos/$repo/git/ref/tags/$tag" --jq '.object.type')
  if [ "$type" = "tag" ]; then
    gh api "repos/$repo/git/tags/$tag_sha" --jq '.object.sha'
  else
    echo "$tag_sha"
  fi
}
```

### Task 15: Create GitOps Deploy Workflow

**Files:**
- Create: `.github/workflows/deploy.yml`

The deploy workflow fires on push to `main`. It uses `tailscale/github-action` to join the tailnet, then runs the Ansible playbook against Minerva's Tailscale IP.

Secrets needed in GitHub repo settings:
- `VAULT_PASSWORD` (already exists for CI)
- `SSH_PRIVATE_KEY` — private key for the `ansible` user
- `TAILSCALE_OAUTH_CLIENT_ID` — Tailscale OAuth client ID
- `TAILSCALE_OAUTH_CLIENT_SECRET` — Tailscale OAuth client secret
- `MINERVA_TAILSCALE_IP` — Minerva's Tailscale IP (e.g. `100.x.x.x`)

- [ ] **Step 1: Get current commit SHA for tailscale/github-action@v2**

Use the same annotated-tag dereference pattern from Task 11 Step 1:

```bash
get_action_sha "tailscale/github-action" "v2"
```

Record the SHA. Also confirm the SHAs for `actions/checkout@v4` and `actions/setup-python@v5` from Task 11 — you'll need all three for this file.

- [ ] **Step 2: Create `.github/workflows/deploy.yml`**

```yaml
---
name: Deploy

on:
  workflow_run:
    workflows: ["CI"]
    types: [completed]
    branches: [main]

permissions:
  contents: read

jobs:
  deploy:
    name: Deploy to Minerva
    runs-on: ubuntu-24.04
    # Only run when CI completes successfully
    if: github.event.workflow_run.conclusion == 'success'
    steps:
      - uses: actions/checkout@34e114876b0b11c390a56381ad16ebd13914f8d5 # v4

      - name: Set up Python
        uses: actions/setup-python@a26af69be951a213d495a4c3e4e4022e16d87065 # v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Install Ansible Galaxy roles
        run: ansible-galaxy install -r ansible/requirements.yml -p ansible/roles

      - name: Install Ansible Galaxy collections
        run: ansible-galaxy collection install community.docker:3.9.0

      - name: Write vault password
        run: echo "${{ secrets.VAULT_PASSWORD }}" > .vault_pass

      - name: Write SSH private key
        run: |
          mkdir -p ~/.ssh
          echo "${{ secrets.SSH_PRIVATE_KEY }}" > ~/.ssh/minerva_ansible
          chmod 600 ~/.ssh/minerva_ansible

      - name: Join Tailscale
        uses: tailscale/github-action@05c58255396c88c609919d571a8b117099c69020 # v2
        with:
          oauth-client-id: ${{ secrets.TAILSCALE_OAUTH_CLIENT_ID }}
          oauth-client-secret: ${{ secrets.TAILSCALE_OAUTH_CLIENT_SECRET }}
          tags: tag:ci

      - name: Run Ansible playbook
        run: |
          ansible-playbook ansible/playbook.yml \
            -i ansible/inventory \
            --private-key ~/.ssh/minerva_ansible \
            -e "ansible_host=${{ secrets.MINERVA_TAILSCALE_IP }}"

      - name: Clean up secrets
        if: always()
        run: |
          rm -f .vault_pass ~/.ssh/minerva_ansible
```

Replace the two `REPLACE_WITH_SHA` placeholders with the actual SHAs from Step 1 and Task 11 Step 1.

The `StrictHostKeyChecking=accept-new` set in `ansible.cfg` (Task 3) handles the ephemeral runner case correctly — it trusts a new host key on first connect. No env var override is needed.
